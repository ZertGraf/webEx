Основное
Разработать автоматизированную информационную систему (АИС) «Электронная библиотека».

Система позволяет автоматизировать ведение реестра книг и читатаелей, осуществлять поиск книг и оставлять рецензии.

Описание книги состоит из следующих полей.

Имя	Тип	Обязательное
название	VARCHAR	Да
краткое описание	TEXT	Да
год	YEAR	Да
издательство	VARCHAR	Да
автор	VARCHAR	Да
объём (в страницах)	INT	Да
Здесь и далее подразумевается, что в СУБД у каждой таблицы будет колонка id типа INT в форме первичного ключа (PRIMARY KEY). Названия полей и таблиц обязательно подберите на английском языке.

У книги может быть один или более жанр, поэтому также в системе должна быть отдельная таблица с жанрами, содержащая поля «идентификатор» и «название» (обязательное, уникальное). Связь между таблицами книг и жанров – «многие ко многим» (т. е. необходимо создать соединительную таблицу).

У каждой книги должна быть обложка (изображение). Информация об обложках должна храниться в отдельной таблице со следующей структурой.

Имя	Тип	Обязательное
название файла	VARCHAR	Да
MIME-тип	VARCHAR	Да
MD5-хэш	VARCHAR	Да
книга	Внешний ключ	Да
У аутентифицированных пользователей должна быть возможность написать рецензию на книгу. Таблица с рецензиями должна содержать следующие поля.

Имя	Тип	Обязательное
книга	Внешний ключ	Да
пользователь	Внешний ключ	Да
оценка	INT	Да
текст	TEXT	Да
дата добавления	TIMESTAMP	Да
Поле «дата добавления» должно заполняться автоматически.

Пользователи описываются следующими полями.

Имя	Тип	Обязательное
логин	VARCHAR	Да
хэш пароля	VARCHAR	Да
фамилия	VARCHAR	Да
имя	VARCHAR	Да
отчество	VARCHAR	Нет
роль	Внешний ключ	Да
Роли пользователей представлены в отдельной таблице следующего вида.

Имя	Тип	Обязательное
название	VARCHAR	Да
Описание	TEXT	Да
Список ролей:

администратор (суперпользователь, имеет полный доступ к системе, в том числе к созданию и удалению книг)
модератор (может редактировать данные книг и производить модерацию рецензий)
пользователь (может оставлять рецензии).
Неаутентифицированные пользователи могут обращаться только к главной странице, странице входа и странице просмотра информации о книге. При попытке открытия других страниц должна происходить переадресация на страницу входа с выводом flash-сообщения «Для выполнения данного действия необходимо пройти процедуру аутентификации».

Жанры, пользователи и роли заполняются в СУБД и не требуют создания дополнительной админки.

Базовый шаблон
Базовый шаблон содержит элементы, отображающиеся на всех типовых страницах проекта.

Панель навигации
Состоит из верхней горизонтальной полосы навигации, в которой находятся кнопки или текстовые ссылки для перехода по разделам. В правой части для неаутентифицированных пользователей отображается кнопка «Войти», для аутентифицированных – ФИО и кнопка «Выйти». По нажатию на кнопку выйти происходит разлогинивание пользователя и переадресация на текущую или главную страницу.

Авторство
Подпишите работу: в любой части шаблона укажите номер группы и ФИО.

Главная страница
Список книг
На главной странице отображается список книг, отсортированный по дате выхода (сначала новые). Для каждой книги должно быть отображено название, жанр(ы), год, средняя оценка пользователей, количество рецензий, оставленных пользователями на эту книгу. Формат отображения выбирается на усмотрение студента (это может быть таблица, «плитка» и т. д.).

На главной странице должна быть реализована пагинация. На одной странице должно отображаться не более 10 книг.

Права доступа
Напротив каждой книги в зависимости от роли текущего пользователя должны отображаться кнопки (значки) для просмотра данных о книге, редактирования, удаления.

Кнопка «Просмотр» должна быть доступна всем пользователям. Кнопка «Редактирование» – только пользователям с ролями «Администратор» и «Модератор». Кнопка «Удалить» – только пользователям с ролью «Администратор».

Под списком книг для пользователей с ролью «Администратор» должна отображаться кнопка «Добавить книгу».

Если пользователь не аутентифицирован, то при попытке доступа к странице, недоступной неаутентифицированным пользователям, он должен быть перенаправлен на страницу входа с сообщением «Для выполнения данного действия необходимо пройти процедуру аутентификации».

В случае, если пользователь попробует произвести действие, для выполнения которого у него недостаточно прав (например, введёт в адресной страке адрес страницы добавления книги), то он должен быть перенаправлен на главную страницу с сообщением «У вас недостаточно прав для выполнения данного действия».

Удаление записей
По нажатию на кнопку «Удалить» должно открываться модальное окно с заголовком «Удаление книги». В модальном окне должно быть сообщение «Вы уверены, что хотите удалить книгу {название книги}?» и кнопки «Да» и «Нет». По нажатию на кнопку «Нет» происходит закрытие модального окна, а по нажатию на кнопку «Да» отправляется запрос на сервер для удаления записи из базы. После обработки заброса пользователь должен быть перенапрвлен на главную страницу, на которой должно отобразиться сообщение об успешном удалении записи (flash-сообщение). При удалении книги должны удаляться все связанные с ней записи (обложка, рецензии, записи из соединительной таблицы книги-жанры). Для этого нужно при создании соответствующих внешних ключей прописать ON DELETE CASCADE и не забыть удалить из файловой системы файл, соответствующий обложке удаляемой книги.

Страница добавления/редактирования книги
На страницах добавления и редактирования книги должна располагаться форма ввода данных о книге. Она должна содержать набор полей, соответствующих полям таблицы с книгами. Так как набор полей у форм добавления и редактирования практически совпадает, для их реализации должен быть использован макрос. Единственное отличие форм создания и редакирования – это отсутствие поля для загрузки файла с обложкой на странице редактирования. Предполагается, что обложка загружается при создании записи и не подлежит редактированию.

На форме для добавления книги должно быть отдельное поле для загрузки файла с обложкой.

Замечание
Для передачи файлов на сервер не забудьте проставить атрибут enctype="multipart/form-data" у тега формы!
В поле с описанием книги должна быть возможность использовать язык разметки Markdown. Рекомендуется для реализации этого поля использовать тег textarea и библиотеку EasyMDE, добавляющую к текстовому полю дополнительные элементы управления, облегчающие форматирование текста.

Для реализации поля «Жанр» рекомендуется использовать тег select с проставленным атрибутом multiple (мультиселект).

Перед сохранением описания книги необходимо «прогнать» введённое пользователем значение через санитайзер (например, Bleach), чтобы экранировать все потенциально опасные теги (например, скрипты), которые мог ввести пользователь.

При нажатии на кнопку «Сохранить», на сервер должен быть отправлен запрос для сохранения данных. В случае, если при сохранении данных возникла ошибка, пользователю должна отобразиться форма с введёнными данными и сообщение «При сохранении данных возникла ошибка. Проверьте корректность введённых данных.». В случае успешного сохранения пользователь должен быть перенаправлен на страницу просмотра данных о книге.

Если в транзакции присутствует более одного запроса, то ошибка может возникнуть при выполнении любого из них. В этом случае необходимо позаботиться о том, чтобы откатить изменения, которые уже были внесены в базу в рамках текущей транзакции (т. е. до возникновения ошибки). Для этого в блоке except нужно выполнить mysql.connection.rollback() (если вы используете mysql-connector-python) или db.session.rollback() (если вы используете SQLAlchemy).

Для предотвращения дублирования файлов при сохранении изображения необходимо вычислить его MD5-хэш. Перед тем как добавлять новую запись в таблицу с изображениями, необходимо проверить, нет ли в ней записи с таким же значением хэша, как для текущего изображения. Если такая запись найдена, значит это изображение уже было загружено ранее, и нет необходимости заново его сохранять -- можно взять идентификатор найденной записи. В противном случае необходимо добавить новую запись в таблицу и сохранить файл в файловой системе. В качестве названия файла рекомендуется использовать идентификатор записи в таблице с изображениями (так как пользовательские названия файлов могут совпадать для разных файлов).

Замечание
Необходимо учитывать, что сохранение книги нужно выполнить перед сохранением обложки, так как при сохранении обложки нужно указать идентификатор книги. Для того, чтобы получить этот идентификатор, можно воспользоваться свойством lastrowid объекта cursor (подробнее см. в документации).

Сохранение файла с обложкой в файловую систему лучше производить после того, как будут добавлены записи в таблицы с изображениями и книгами, так как в таком случае не придётся удалять файл в случае возникновения ошибки при добавлении записей в СУБД.

Если при загрузке файла с обложкой ваше приложение падает с ошибкой, посмотрите логи веб-сервера с помощью команды sudo tail -f /opt/unit/unit.log. Если вы увидите ошибку [alert] ... mkstemp(/opt/unit/tmp/...) failed (2: No such file or directory), нужно создать директорию для хранения временных файлов sudo mkdir /opt/unit/tmp, а затем изменить права доступа к этой директории sudo chmod -R a=rwx,o+t /opt/unit/tmp. Последняя команда также должна помочь, если у вас возникает ошибка [alert] ... mkstemp(/opt/unit/tmp/...) failed (13: Permission denied).

Страница просмотра данных о книге
На странице просмотра должна отображаться обложка книги, а также все данные о книге. Описание книги должно быть преобразовано из Markdown в HTML и подставлено на страницу. Формат отображения на усмотрения студента.

Внизу страницы должны быть отображены все рецензии, оставленные пользователями к этой книге. Для каждой рецензии должен быть указан пользователь, который её оставил, оценка и текст рецензии (с учётом форматирования).

Функциональность создания рецензий
Если пользователь аутентифицирован, имеет роль «Пользователь», «Модератор» или «Администратор», и не писал ранее рецензию на данную книгу, то для него внизу страницы должна отображаться кнопка «Написать рецензию». Если он уже писал рецензию на данную книгу, то вместо кнопки ему должна быть отображена его рецензия.

По нажатию на кнопку «Написать рецензию» пользователь должен попасть на страницу с формой для создания рецензии. В ней должны быть поля «Оценка» и «Текст рецензии».

Поле оценки представляет собой селектор с опциями

5 – отлично (значение по умолчанию),
4 – хорошо,
3 – удовлетворительно,
2 – неудовлетворительно,
1 – плохо,
0 – ужасно.
Пользователю должны отображаться только названия оценок, а на сервер должны передаваться только числовые значения.

Поле «Текст рецензии» должно представлять с собой тег textarea. В тексте рецензии должна быть реализована поддержка языка форматирования Markdown.

По нажатию на кнопку «Сохранить», если сохранение выполнено успешно, пользователь должен быть перенаправлен на страницу просмотра книги. В противном случае ему должно быть отображено сообщение об ошибке, и он должен остаться на странице с формой создания рецензии.

При сохраненни текста рецензии необходимо не забыть «пропустить» его через санитайзер.

Страница входа
Отображает форму ввода логина и пароля с кнопкой «Войти» и чекбоксом «Запомнить меня». В случае неверных логина и/или пароля выводит сообщение «Невозможно аутентифицироваться с указанными логином и паролем», предлагается заново заполнить форму.

Вариант 1. Функциональность модерации рецензий
Добавьте отдельную таблицу со статусами рецензий. Возможные статусы:

на рассмотрении,
одобрена,
отклонена.
Добавьте в таблицу с рецензиями обязательное поле «статус» (внешний ключ). При создании новой рецензии ей должен автоматически проставляться статус «На рассмотрении». На странице книги должны быть отображены только рецензии со статусом «Одобрено».

Для пользователей с ролью «Пользователь» в панели навигации должен отображаться раздел «Мои рецензии», при переходе в который пользователю должен быть отображён список всех его рецензий с указанием статуса, книги, оценки и текста рецензии.

Для пользователей с ролью «Модератор» в панели навигации должен отображаться раздел «Модерация рецензий», в котором должна отображаться таблица с рецензиями в статусе «На рассмотрении», отсортированными по дате добавления. В таблице должны присутствовать название книги, имя пользователя и дата добавления рецензии. Должна быть реализована пагинация.

Напротив каждой строки в таблице должна быть кнопка «Рассмотреть», по нажатию на которую открывается страница с данными рецензии и кнопками «Отклонить» и «Одобрить». При нажатии на первую из них статус рецензии меняется на «Отклонено» и пользователь перенаправлятся к списку рецензий, подлежащих модерации. По нажатию на вторую кнопку -- статус меняется на «Одобрено».

Вариант 2. Функциональность формирования подборок книг
Добавьте отдельную таблицу с подборками книг (в этой таблице должны быть поля «идентификатор», «название» и «пользователь»). Также добавьте связующую таблицу «многие ко многим» между таблицами с книгами и подборками.

Для пользователей с ролью «Пользователь» в панели навигации должен отображаться раздел «Мои подборки». В данном разделе должна отображаться таблица с подборками пользователя. В таблице должно быть приведено название подборки и количество книг, которые в неё входят. Напротив каждой строки должна быть кнопка «Просмотр». По нажатию на эту кнопку пользователю должна быть отображена страница со списком книг, включённых в эту подборку. Формат отображения на усмотрение студента.

Внизу страницы должна быть кнопка «Добавить подборку», по нажатию на которую должно открываться модальное окно с заголовком «Новая подборка» и текстовым полем для ввода названия подборки. Внизу модального окна должны быть кнопки «Добавить» и «Отмена». По нажатию на кнопку «Отмена» модальное окно должно закрываться, а по нажатию на кнопку «Добавить» должен отправляться запрос на сервер, в результате обработки которого в БД добавится новая подборка, после чего пользователь должен быть перенаправлен на страницу «Мои подборки» и ему должно отобразиться сообщение об успешном добавлении подборки.

Для пользователей с ролью «Пользователь» на странице просмотра книги должна отображаться кнопка «Добавить в подборку». По нажатию на эту кнопку пользователю должно быть открыто модальное окно с заголовком «Добавить в подборку». В модальном окне должен быть селектор, предлагающий выбрать одну из ранее созданных данным пользователем подборок. Внизу окна должны быть кнопки «Добавить» и «Отмена». По нажатию на кнопку «Отмена» модальное окно должно закрываться, а по нажатию на кнопку «Добавить» должен отправляться запрос на сервер, в результате обработки которого выбранная книга должна быть добавлена в выбанную подборку, после чего пользователь должен быть перенаправлен на страницу просмотра книги, и ему должно отобразиться сообщение об успешном добавлении книги в подборку.

Вариант 3. Функциональность поиска книг
Добавьте на главную страницу (со списком книг) форму для поиска книг. В форме должны быть представлены следующие поля:

название (текстовое),
жанр (мультиселект),
год (мультиселект),
объём от/до (текстовое),
автор (текстовое).
По нажатию на кнопку «Найти» должен производиться поиск книг по введённым значениям. Если поле формы не заполнено, то его не нужно учитывать при фильтрации записей.

При фильтрации по названию и автору нужно учитывать частичное совпадение.

Варианты выбора в поле «год» должны формироваться исходя из содержимого БД.

При переходе на следующую страницу фильтрация не должна сбрасываться.

Вариант 4. Функциональность учёта истории посещений
Добавьте на главную страницу системы раздел «Популярные книги». В этом разделе должны отображаться 5 наиболее просматривмых за последние 3 месяца книг (формат отображения на усмотрение студента, но обязательно должны присутствовать ссылки на страницы просмотра книг). Счётчик посещений страниц книг должен обновляться автоматически при каждом просмотре страницы. При этом, чтобы избежать злоупотреблений, для одного пользователя должно учитываться не более 10 посещений страницы конкретной книги в день (все посещения сверх этого порога не должны сохраняться).

Добавьте на главную страницу раздел «Недавно просмотренные книги», в котором должны отображаться 5 книг, которые просмотрел данный пользователь (в том числе и неаутентифицированный пользователь) в обратном хронологическом порядке – по убыванию даты и времени просмотра. Если пользователь ещё не просмотрел ни одной книги – данный раздел должен быть скрыт.

Для пользователей с ролью «Администратор» в навигационной панели должен быть доступен пункт «Статистика». При переходе по данной ссылке должна открываться страница с двумя вкладками: «Журнал действий пользователей» и «Статистика просмотра книг».

На вкладке «Журнал действий пользователей» должна отображаться таблица со следующими полями: № (порядковый номер записи), ФИО пользователя (если пользователь не вошёл в систему – «Неаутентифицированный пользователь»), название просмотренной книги, дата и время просмотра. Записи в таблице должны быть отсортированы в порядке убывания даты. Также необходимо реализовать постраничное отображение записей – не более 10 записей на страницу.

На вкладке «Статистика просмотра книг» должна отображаться таблица со статистикой просмотров: № (порядковый номер записи), название книги, количество просмотров. Записи должны быть отсортированы по убыванию количества просмотров. Количество просмотров подсчитывается только на основе данных аутентифицированных пользователей. Также необходимо реализовать постраничное отображение записей – не более 10 записей на страницу. Над таблицей должна располагаться форма, позволяющая выбрать промежуток времени, за который нужно подсчитать статистику (поля «дата от» и «дата до» (включительно), кнопка «применить»). Пользователь может заполнить только начало или конец диапазона.

На обеих вкладках («Журнал действий пользователей» и «Статистика просмотра книг») должна располагаться кнопка «Экспорт в CSV», по нажатию на которую на компьютер пользователя должен быть загружен файл с соответствующими данными в формате CSV. В загружаемый файл должны войти все данные (а не только данные текущей страницы). Название файла должно быть осмысленным и содержать дату генерации файла.
